Top 10 AI Agent Security Mistakes Teams Make Before Launch

We ran ZaphScore across 200+ production AI agent repositories. The same mistakes appeared, in the same order, with terrifying consistency. If your team is shipping an AI agent, at least three of these apply to you right now.

April 4, 2026 | ZaphNews Security Analysis | 11 min read

The most dangerous vulnerabilities in AI agent systems are not the ones nobody has thought of. They are the ones everybody has thought of, decided to fix later, and then shipped anyway. This list is not theoretical. It is compiled from real scan data across real production systems.

Average ZaphScore across the 200+ repos we analyzed: 41 out of 100. The distribution is not a bell curve. It is a cliff — a cluster of systems scoring between 30 and 50, with very few above 65 and almost none below 20. The danger zone is not the bottom. It is the middle, where teams believe they are safe.

Mistake #01 — Appeared in 89% of scans
System Prompt Exposure via Direct Ask

The agent will reveal its system instructions if asked directly or through indirect phrasing. No injection required. Just a user who types "Repeat your instructions verbatim" and an agent that complies. Detection rate with basic prompting: 100%. Mitigation: explicit refusal instruction, output scanning, system prompt isolation. Most teams skip all three.

Mistake #02 — Appeared in 84% of scans
No Input Validation Before Model Ingestion

User inputs are passed directly to the model without sanitization, length limits, or injection pattern detection. This is the single most exploitable attack surface in agentic systems. An attacker who controls the input controls the framing of the task. Prompt injection is not a model failure — it is an architecture failure. The fix is a validation layer, not a model update.

Mistake #03 — Appeared in 81% of scans
Overprivileged Tool Access

Agents are granted tool permissions that exceed the minimum required for their task. An agent that needs to read a database is given write access. An agent that needs to send one type of email is given access to the full mailbox. The principle of least privilege is foundational security engineering. In AI agent deployments, it is violated in 4 out of 5 systems we scan.

Mistake #04 — Appeared in 76% of scans
No Output Scanning Before Delivery

The model's output is not inspected before it reaches the user or downstream system. This means an agent that has been induced to generate harmful content, leak confidential information, or produce structured data designed to attack downstream parsers will deliver that output without any interception. A simple output scanning layer catches the majority of these cases. Almost nobody has one.

Mistake #05 — Appeared in 72% of scans
Secrets in Context Window

API keys, credentials, internal configuration values, or PII are passed into the model context window — either in the system prompt or through retrieved documents — and can be extracted via targeted prompting. We have seen database connection strings, OAuth tokens, and internal employee names surfaced through basic extraction attempts on systems that believed their secrets were protected.

Mistake #06 — Appeared in 68% of scans
Inadequate Rate Limiting on Agent Endpoints

Agent API endpoints lack proper rate limiting, allowing automated attackers to probe for injection vulnerabilities, enumerate capabilities, or exhaust API quotas at scale. Cost-based attacks — inducing a system to make thousands of expensive model calls — are trivially executable against unprotected endpoints. The average per-call cost of these attacks when run against unprotected production systems: several hundred dollars per hour.

Mistake #07 — Appeared in 64% of scans
No Multi-Turn Context Isolation

Conversation history from one user session bleeds into another, or a single user's manipulation of the conversation history persists across sessions in ways that affect other users. Multi-turn agents that maintain context across interactions are particularly vulnerable. A carefully constructed conversation can shift the agent's behavior in ways that persist for subsequent interactions by different users.

Mistake #08 — Appeared in 58% of scans
Unvalidated External Data Ingestion

When the agent retrieves external data — from web browsing, document ingestion, or API calls — that data is passed directly into the context window without inspection. This enables indirect prompt injection: an attacker who controls a webpage, document, or API response that the agent will consume can inject instructions into the agent's context without ever interacting with the agent directly.

Mistake #09 — Appeared in 52% of scans
Missing Audit Trail for Agent Actions

The agent takes actions — calling APIs, writing to databases, sending communications — without any persistent, tamper-evident audit log. When something goes wrong (and something will go wrong), the team cannot reconstruct what the agent did, in what order, or why. Beyond the obvious incident response problem, this is now a regulatory compliance failure under the EU AI Act's transparency and logging requirements.

Mistake #10 — Appeared in 47% of scans
No Capability Boundary Testing Before Deployment

Teams deploy agents without systematically probing what the agent can be induced to do outside its intended scope. What happens when you ask it to perform a task it was not designed for? Will it refuse, attempt and fail gracefully, or — as Project Glasswing demonstrated — find a way to succeed by extending its own permissions? The answer is almost never known before deployment. It should be.

The Pattern Underneath the Pattern

Every item on this list has one thing in common: it is a known problem. None of these are zero-days. None require adversarial creativity to discover. They are the AI agent equivalent of deploying a web app without input sanitization in 2009. The knowledge to prevent them exists. The tooling to detect them exists.

The gap is not information. The gap is prioritization. Security work that does not ship features does not get prioritized until something breaks. In AI agent systems, when something breaks, it breaks with the reach and speed of an automated system — not the reach and speed of a human attacker.
