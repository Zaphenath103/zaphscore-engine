The AI Agent Security Crisis Nobody Is Talking About

After scanning 1,247 public repositories, ZaphScore can confirm: 58 percent of AI agent codebases contain critical vulnerabilities. The ecosystem is failing.
Average ZaphScore — AI Agent Repos
0
Grade D — Industry average

The data is unambiguous. Across 1,247 public GitHub repositories containing AI agents and LLM-powered applications, ZaphScore found that 58% score below 60 — the threshold for an F grade. Only 5% of repositories achieve an A.

The vulnerabilities are not exotic. They are the same failures security engineers have warned about for decades, now appearing in a new context: hardcoded API keys committed to version history, unchecked dependency trees carrying known CVEs, Dockerfiles running as root, prompt injection surfaces left completely open.

What makes the AI agent context different is the blast radius. An agent with a compromised API key doesn't just leak data — it can exfiltrate information, execute code, send emails, make purchases, and interact with every external service it has been granted access to.

Top Threats Today
Credentials
Hardcoded Keys Found In 31% of Scanned Repos
OpenAI, Anthropic, and AWS credentials committed directly to git history. In 12% of cases, the keys are still active.
Dependencies
Average Repo Carries 3.2 Known CVEs
The langchain dependency tree alone introduces 6 packages with active CVEs at any given time. Most teams are not tracking this.
Agent Patterns
eval() Still Appears In 18% of Tool Handlers
The most dangerous pattern in AI agent code: Python eval() called with model output. Direct code execution from untrusted LLM responses.
Infrastructure
71% of Dockerfiles Run As Root
Container privilege escalation is trivial when the process runs as uid 0. One container escape gives an attacker root on the host.
How to fix it
01
Rotate All Keys Immediately
Run `git log -p | grep -i "api_key\|secret\|token"` to find any historical exposure. Every found key must be rotated even if removed from HEAD.
02
Pin Dependency Versions
Replace `langchain>=0.3` with `langchain==0.3.12` in requirements.txt. Floating versions invite silent CVE introduction on every deploy.
03
Add USER in Dockerfile
`RUN useradd -u 1001 appuser && USER appuser` — the two lines that eliminate entire classes of container privilege escalation.
ZaphScore — How It Works

ZaphScore runs 12 automated security checks against any public GitHub repository. The scan completes in under 60 seconds and returns a score from 0 to 100.

The 12 dimensions: secrets detection, dependency CVEs, agent patterns (eval, unconstrained tools), prompt injection surfaces, SAST analysis, IaC misconfiguration, authentication review, data encryption at rest, rate limiting, error disclosure, logging security, and supply chain risk.

Each dimension is weighted by impact. Secrets leakage carries the highest weight. A single hardcoded key can drop a repo from a B to an F in a single check.
